### 2. Жёсткие блокировки вместо ошибок

**Системы (Сет 1/2/3 по ширине створки):**
- В выпадающем списке систем для каждой створки скрывать опции, чьё `minSashWidth` больше текущей ширины створки

- Убрать соответствующую error-проверку из `calculate.ts` (фильтрация в UI её заменяет)

**Стекло Мору/Кафедрал/Вижн (макс. высота створки 2800):**
- На полях Input «Высота проёма» — динамический атрибут `max`: если выбрано стекло с `maxHeight`, то `max = maxHeight − sashHeightOffset` (для текущего типа)
- Подсказка под полем: «Макс. высота для выбранного стекла: X мм»
- При выборе стекла, у которого текущая высота уже превышает лимит, обрезать `openingHeight` до допустимого
- Убрать error-проверку из `calculate.ts`

### 3. Добавить «Расположение ручек» (позиции 1–4)

В ТЗ схема:
```
2  3
1  4
```
- Для каждой створки, у которой `hasHandle: true`, показать визуальный выбор позиций (4 чекбокса в виде сетки 2×2)
- Хранить в state: `handlePositions: number[][]` — массив выбранных позиций для каждой створки
- Кол-во ручек не привязываю автоматически к сумме позиций (в ТЗ это отдельный селект), но валидирую: общая сумма позиций по всем створкам должна быть ≥ `handleCount`. Если меньше — предупреждение.
- В summary заказа добавить строку «Створка N: ручки в позициях 1, 3»

### 4. Визуальная проверка всех 10 типов

После правок прокликать через browser-tool каждый тип (`/configurator/type-1` … `type-7`), убедиться что:
- Списки систем/открываний соответствуют ТЗ для каждой створки
- Максимальное кол-во ручек верное (Тип №1 — до 2, Тип №6 — до 2, остальные — до 4)
- Расчёт меняется корректно при смене параметров
- Жёсткие блокировки работают (попробовать стекло Мору + большую высоту, Сет 2 + узкую створку)
